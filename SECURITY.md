# حالة الأمان

تم تطبيق `supabase/security-hardening.sql` على مشروع Supabase والتحقق منه في 20 حزيران 2026. أعاد فحص الجدول والعمود والدالة وسياسة التخزين والحاوية الخاصة القيم `true` جميعاً.

## مطبّق في الشيفرة

- لا يوجد مفتاح سري في الواجهة؛ المفتاح المستخدم Publishable فقط.
- RLS يعزل طلبات العملاء، وحاوية الملفات خاصة، وروابط الملفات مؤقتة لمدة 60 ثانية.
- مسار التخزين يتحقق من صاحب الطلب، والحد خمسة ملفات و10MB للملف.
- الأنواع المسموحة محصورة في PDF وDOCX وXLSX وCSV وJPEG وPNG وTIFF وKML وKMZ؛ أزيلت ZIP وDOC/XLS القديمة و`application/octet-stream`.
- حد قاعدة البيانات خمسة طلبات للحساب في الساعة.
- إجراءات المدير الحساسة تُقيّد بـ AAL2 بعد ربط TOTP، وتغييرات الحالة والحذف تسجل في `admin_audit_logs`.
- سياسة احتفاظ: 90 يوماً بعد اكتمال الطلب، مع دالة حذف خادمية مجهزة في `RETENTION.md`.
- مكتبة Supabase مثبتة محلياً بالإصدار `2.108.2`، SHA-256: `F06EA3E7D7AB62CCFA27278D02B42D0AABC6CA72DB782C89A66E166516E5C6E8`.
- الخادم المحلي مقيد بـ `127.0.0.1` ويرسل CSP و`nosniff` وDENY للإطارات وسياسة referrer مشددة.

## ممنوعات

- لا تضع `sb_secret_...` أو `service_role` أو مفتاح Turnstile السري في HTML أو JavaScript.
- لا تستقبل كلمات مرور أو رموز دخول من المستخدمين يدوياً.
- لا تجعل حاوية `request-files` عامة، ولا تستخدم `file://` للمصادقة.

## متطلبات ما قبل الإطلاق العام

- نشر الموقع عبر HTTPS وتحديث Site URL وRedirect URLs.
- تفعيل Turnstile وSMTP مخصص من Supabase، ثم تفعيل TOTP لحساب المدير.
- نشر وجدولة دالة الحذف بعد أخذ نسخة احتياطية تجريبية.
- اختبار RLS بحساب عميل مستقل، وتجربة الاستعادة من نسخة احتياطية.
- الملفات ليست مفحوصة آلياً ضد البرمجيات الضارة؛ لا تفتح ملفاً غير متوقع، وأضف خدمة فحص قبل توسيع المنصة لاستقبال عام واسع.
